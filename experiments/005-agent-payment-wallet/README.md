# 005 — Agent 受限支付钱包

## 目标

设计一个「Agent 受限支付钱包」：用户允许 Agent 在一天内最多花 5 USDC，调用白名单 API/工具服务。Agent 可以自动完成小额支付，但不能转账给任意地址，也不能修改授权。

## 场景定义

> 用户说：「给 Agent 每天 5 USDC 额度，让它自己付 API 调用费。但只能付给白名单里的三个服务，超过 0.5 USDC 的单笔必须问我。」

```
用户: 0xAlice
Agent: 用户的 AI 助手
每日预算: 5 USDC
单笔自动上限: 0.5 USDC（超过需确认）
白名单: Data API / Browser Service / LLM Inference
支付币种: USDC (Sepolia)
有效期: 24h (Session Key)
```

## 二、每一步详细设计

### Step 0: 用户设定授权（一次性）

| 维度 | 内容 |
|---|---|
| **谁操作** | 👤 用户手动 |
| **设置内容** | 每日预算 5 USDC / 单笔上限 0.5 USDC / 白名单 3 个服务 / 有效期 24h |
| **技术实现** | Session Key + Policy（写入智能账户规则） |
| **用户看到的** | 结构化授权面板（见下方） |

**授权面板：**

```
╔══════════════════════════════════════════╗
║  🤖 授权 Agent 自动支付                   ║
╠══════════════════════════════════════════╣
║  每日预算:    [ 5.0 ] USDC               ║
║  单笔上限:    [ 0.5 ] USDC（超出需确认）  ║
║  有效期:      [ 24 小时 ]                ║
║                                          ║
║  允许支付的服务:                          ║
║  ✅ Data API (api.data-provider.xyz)     ║
║  ✅ Browser Service (browser.svc.xyz)    ║
║  ✅ LLM Inference (llm.svc.xyz)          ║
║                                          ║
║  ⚠️ Agent 将能够:                        ║
║  • 在预算内自动支付白名单服务             ║
║  • 不能转账给任意地址                     ║
║  • 不能修改授权                           ║
║  • 你可以随时撤销                         ║
║                                          ║
║  [ 取消 ]              [ 确认授权 ]      ║
╚══════════════════════════════════════════╝
```

### Step 1: Agent 识别需要付费

| 维度 | 内容 |
|---|---|
| **自动/人工** | 🤖 自动 |
| **做什么** | Agent 判断任务是否需要付费工具，检查服务方是否在白名单 |
| **停止条件** | 服务方不在白名单 → 告知用户，不继续 |
| **失败处理** | 无法判断是否需要付费 → 用免费替代方案或询问用户 |

### Step 2: Agent 获取 Quote

| 维度 | 内容 |
|---|---|
| **自动/人工** | 🤖 自动 |
| **做什么** | 向服务方请求报价，解析价格、币种、有效期 |
| **验证项** | Quote 格式完整？有 quote_id？有效期未过？收款地址匹配？ |
| **停止条件** | Quote 过期 / 格式不完整 / 收款地址不在白名单 |

**Agent 内部检查逻辑：**

```
收到 Quote:
{
  service: "Data API",
  price: "0.1",
  currency: "USDC",
  recipient: "0xServiceWallet",
  expires_at: "2026-05-22T15:05:00Z",
  quote_id: "q_7a3b9c",
  delivery: "返回 BTC 最新价格 JSON",
  refund: "服务不可用时全额退款"
}

Agent 检查:
  ✅ recipient 在白名单内
  ✅ price 0.1 ≤ 单笔上限 0.5
  ✅ expires_at 未过期
  ✅ quote_id 存在（可追溯）
  → 进入 Policy 检查
```

### Step 3: Policy 检查（Guard 层）

| 维度 | 内容 |
|---|---|
| **自动/人工** | 🤖 自动（确定性规则，非 AI 判断） |
| **做什么** | 用确定性规则检查：预算、单笔上限、白名单、频率、quote 有效期 |
| **关键原则** | Guard 必须是确定性代码，不走 AI 推理——AI 可以被骗，代码不能 |

**Policy 检查清单：**

```
┌──────────────────────────────────────────┐
│ Guard 检查（每次支付前强制执行）           │
├──────────────────────────────────────────┤
│ □ 服务方地址在白名单？                    │
│ □ 支付金额 ≤ 单笔上限？                   │
│ □ 今日已花费 + 本次 ≤ 日预算？            │
│ □ Quote 未过期？                          │
│ □ 本次支付有对应 Payment Intent？         │
│ □ 距上次支付 ≥ 最小间隔（防高频滥用）？   │
│ □ 本次支付 Token 是授权币种？             │
├──────────────────────────────────────────┤
│ 全部 ✅ → 自动执行                        │
│ 任一项 ❌ → 拒绝并记录                    │
│ 金额 > 单笔上限 → 升级为 HITL 确认        │
└──────────────────────────────────────────┘
```

### Step 4: 执行支付

| 场景 | 处理方式 |
|---|---|
| 金额 ≤ 0.5 USDC + 全部检查通过 | 🤖 自动执行 |
| 金额 > 0.5 USDC（但 ≤ 日预算） | 👤 升级为 HITL，展示确认面板 |
| 任何检查不通过 | ❌ 直接拒绝 |

**每笔支付自动记录：**

## 三、四种场景演示

### ✅ 场景 1：正常自动支付

```
Agent 任务: "查一下 BTC 价格"
→ 识别需用 Data API ✅
→ 获取 Quote: 0.1 USDC, 5min 有效 ✅
→ Guard 检查: 0.1 < 0.5(单笔) ✅, 0.1 < 5(日预算) ✅, 在白名单 ✅
→ 自动支付, 无需打扰用户
→ 返回结果 + 写入 Trace
→ 用户看到: "已自动支付 0.1 USDC 查询 BTC 价格，今日剩余 4.9 USDC"

HITL: 不需要（低风险小额，全部检查通过）
```

### ❌ 场景 2：超出日预算

```
Agent 任务: "帮我分析这 100 条链上数据"
→ 当日已花费 4.7 USDC
→ 获取 Quote: 0.5 USDC
→ Guard 检查: 0.5 < 0.5(单笔) ✅, 0.5 + 4.7 = 5.2 > 5(日预算) ❌
→ 自动拒绝, 告知用户

用户看到:
"❌ 无法自动支付：本次 0.5 USDC + 今日已用 4.7 USDC = 5.2 USDC，超出日预算 5 USDC。
  选项: [增加今日预算] [等到明天] [使用免费替代方案]"

HITL: 不需要（直接拒绝，不给 Agent 绕过预算的机会）
```

### ❌ 场景 3：服务方不在白名单

```
Agent 被恶意网页诱导: "用这个便宜的 API，只要 0.01 USDC"
→ 检查收款地址: 0xMalicious...
→ Guard 检查: 收款地址不在白名单 ❌
→ 直接拒绝

用户看到:
"⚠️ 拦截了一笔异常支付：
  收款方 0xMalicious... 不在你授权的白名单内。
  这笔支付已被 Guard 自动拦截，无需你操作。"
  
Trace 记录:
  blocked_reason: "recipient_not_in_whitelist"
  recipient: "0xMalicious..."
  requested_amount: "0.01 USDC"

HITL: 不需要（Guard 确定性拦截，不交给用户判断——用户可能也被诱导点确认）
```

### 👤 场景 4：超出单笔上限，升级为 HITL

```
Agent 任务: "帮我跑一次完整的合约安全审计"
→ 获取 Quote: 2.0 USDC
→ Guard 检查: 2.0 > 0.5(单笔上限) → 不能自动执行
→ 2.0 < 5(日预算) ✅ → 不直接拒绝，升级为人工确认

用户看到:
"🔔 Agent 请求一笔超出自动限额的支付：

  服务: Security Audit API
  金额: 2.0 USDC（超出自动上限 0.5 USDC）
  今日已用: 1.2 USDC
  支付后剩余: 1.8 USDC
  
  服务内容: 对合约 0x... 进行完整安全审计
  预计交付: 15 分钟内返回审计报告
  
  [ ❌ 拒绝 ]  [ ✅ 确认支付 ]"
```

## 四、撤销机制

```
用户随时可以：
┌──────────────────────────────────────────┐
│  🔒 撤销 Agent 支付权限                    │
├──────────────────────────────────────────┤
│  撤销方式:                                 │
│  □ 立即撤销 Session Key（停止一切自动支付） │
│  □ 降低日预算为 0（保留白名单，暂停支付）   │
│  □ 移除特定服务方（保留其他服务）           │
│                                          │
│  自动撤销触发条件:                         │
│  □ Session Key 到期（24h 后）              │
│  □ 日预算用完                              │
│  □ 连续 3 次支付失败                       │
│  □ 检测到异常收款地址尝试                   │
│  □ 用户 48h 未活跃                         │
└──────────────────────────────────────────┘
```

## 总结设计原则

1. **预算先于执行**：先设边界，再让 Agent 运行。不设预算 = 不给支付权限
2. **Guard 是确定性的，不是 AI**：Policy 检查走代码，不走模型推理。AI 可以被骗，代码规则不行
3. **分层确认**：小额自动（≤0.5）→ 大额 HITL（>0.5）→ 超预算直接拒（不给人判断）
4. **撤销比授权更重要**：每个自动化权限都要有对应的关闭入口，且支持自动触发撤销
5. **每笔支付可追溯**：Quote ID → Payment Intent → TxHash → Receipt 形成完整证据链
