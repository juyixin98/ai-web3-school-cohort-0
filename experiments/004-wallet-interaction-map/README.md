# 004 — 钱包交互地图

## 目标

以测试钱包连接测试网 dApp 为例，画出完整交互流程，标注每一步的读/写性质、用户应看到的信息，以及 AI Agent 辅助时的 HITL 要求。

## 场景设定

- 钱包：MetaMask（测试钱包，非主资产钱包）
- 网络：Sepolia 测试网
- dApp：一个测试网 Swap 页面

## 每一步详细拆解

### Step 1: 连接钱包

| 维度 | 内容 |
|---|---|
| **类型** | 📖 只读 |
| **链上变化** | 无 |
| **技术过程** | dApp 调用 `eth_requestAccounts` → MetaMask 弹窗 → 用户选择账户 → 返回地址列表 |
| **gas 消耗** | 0 |

**用户看到的关键信息：**

```
╔══════════════════════════════════════╗
║  MetaMask 弹窗                        ║
╠══════════════════════════════════════╣
║  "这个网站请求连接你的钱包"           ║
║                                      ║
║  网站: app.uniswap.org               ║
║  请求: 查看你的钱包地址和余额         ║
║  账户: 0x1234...abcd                 ║
║                                      ║
║  它将能:                             ║
║  ✅ 查看你的地址                      ║
║  ✅ 查看你的 ETH 余额                 ║
║  ❌ 不能转账                          ║
║  ❌ 不能签名                          ║
║                                      ║
║  [ 取消 ]          [ 连接 ]          ║
╚══════════════════════════════════════╝
```

**AI Agent 辅助时的判断：**

| 场景 | 处理 |
|---|---|
| Agent 自动连接 | ❌ 不可以——连接是授权访问身份，用户必须知道哪个网站要连 |
| Agent 检查是否已连接 | ✅ 可以——只读检查，不影响状态 |
| Agent 建议连接 | ✅ 可以——Agent 说「这个 dApp 需要连接钱包」，用户点确认 |

> 👤 **HITL：必须保留人工确认**。连接 = 暴露身份 + 余额 + 交易历史，这是隐私边界。

---

### Step 2: 检查/切换网络

| 维度 | 内容 |
|---|---|
| **类型** | 📖 只读（检查）/ 📝 可能写（切换） |
| **链上变化** | 无（网络配置在钱包本地，不在链上） |
| **技术过程** | `eth_chainId` 检查 → 不匹配则 `wallet_switchEthereumChain` → 链不存在则 `wallet_addEthereumChain` |

**用户看到的关键信息：**

```
╔══════════════════════════════════════╗
║  MetaMask 弹窗                        ║
╠══════════════════════════════════════╣
║  "这个网站请求切换网络"               ║
║                                      ║
║  当前网络: Ethereum Mainnet           ║
║  请求切换: Sepolia Testnet            ║
║  Chain ID: 11155111                  ║
║  RPC: sepolia.infura.io              ║
║                                      ║
║  [ 取消 ]          [ 切换 ]          ║
╚══════════════════════════════════════╝
```

> ⚠️ 如果网络不存在，还会多一步「添加网络」，用户需要看到 Chain ID、RPC URL、区块浏览器 URL、货币符号。

**AI Agent 辅助时的判断：**

| 场景 | 处理 |
|---|---|
| Agent 检测到网络不对 | ✅ 可以——只读 eth_chainId |
| Agent 建议「请切换到 Sepolia」 | ✅ 可以——提示用户，让用户自己在钱包操作 |
| Agent 自动切换网络 | ❌ 不可以——切换网络可能将用户暴露到恶意 RPC 节点 |
| Agent 自动添加网络 | ❌ 绝对不可以——恶意 RPC 可以伪造链上数据 |

> 👤 **HITL：必须保留人工确认**。切换网络 = 信任新的 RPC 节点。测试网风险低，但主网场景下切换到一个恶意 RPC 节点可能导致签名钓鱼。

---

### Step 3: dApp 读取链上数据

| 维度 | 内容 |
|---|---|
| **类型** | 📖 只读 |
| **链上变化** | 无 |
| **技术过程** | `eth_call` 查询合约（余额、allowance、报价）、`eth_getBalance`、监听区块事件 |

**用户看到的关键信息：**

```
页面自动刷新显示:
┌──────────────────────────────────────┐
│  💰 你的余额                          │
│  0.45 ETH | 1,200 USDC | 800 USDT   │
│                                      │
│  📊 当前报价                          │
│  1 ETH = 3,420 USDC                  │
│  Price Impact: 0.02%                │
│                                      │
│  ⛽ Gas 估算                          │
│  ~$2.80 (est.)                      │
└──────────────────────────────────────┘
```

> 这些都是 `eth_call` + `eth_getBalance`，不消耗 gas，不修改任何状态。

**AI Agent 辅助时的判断：**

| 场景 | 处理 |
|---|---|
| Agent 查询余额/报价/allowance | ✅ 完全可以自动——eth_call 是只读的 |
| Agent 整理数据展示给用户 | ✅ 完全可以——这是 AI 的优势 |
| Agent 定期轮询数据 | ✅ 可以——不消耗 gas |

> 🤖 **HITL：可以完全自动化**。这正是 AI Agent 最擅长的事——读数据、整理、解释。我们之前的 RPC 实验和 eth_call 实验都属于这一步。

---

### Step 4: 签名消息（如登录认证）

| 维度 | 内容 |
|---|---|
| **类型** | ✍️ 签名（链下证明，不在链上记录） |
| **链上变化** | 无（签名在链下完成，不广播交易） |
| **技术过程** | `personal_sign` 或 `eth_signTypedData_v4` |
| **gas 消耗** | 0 |

**用户看到的关键信息：**

```
╔══════════════════════════════════════╗
║  MetaMask 签名请求                    ║
╠══════════════════════════════════════╣
║  账户: 0x1234...abcd                 ║
║                                      ║
║  消息:                               ║
║  "Sign this message to log in to     ║
║   app.uniswap.org"                   ║
║   Nonce: a7f3b9c2                    ║
║                                      ║
║  ⚠️ 签名 ≠ 转账                      ║
║  这不会花费任何 ETH                   ║
║  但可以证明你拥有这个地址              ║
║                                      ║
║  [ 取消 ]          [ 签名 ]          ║
╚══════════════════════════════════════╝
```

> 💡 EIP-712 签名（`eth_signTypedData_v4`）会把数据结构化展示，用户能看到完整的内容而不是一串 hex。登录、授权、投票等场景推荐使用。

**AI Agent 辅助时的判断：**

| 场景 | 处理 |
|---|---|
| Agent 发起签名请求 | ❌ 绝对不可以——签名 = 用户身份的密码学证明 |
| Agent 总结「这条消息是什么意思」 | ✅ 可以——帮用户理解消息内容 |
| Agent 检查签名内容是否合理 | ✅ 可以——但只做辅助提示，不代替用户判断 |

> 👤 **HITL：必须保留人工确认**。虽然签名不消耗 gas、不上链，但它生成一个密码学证明：「这个地址的主人同意这段内容」。恶意签名可以被用在其他地方（如 replay attack），用户必须看懂再签。

### Step 5: 发送交易（如 swap、转账）

| 维度 | 内容 |

| **技术过程** | `eth_sendTransaction` → 广播到 mempool → 矿工打包 → 链上确认 |
| **gas 消耗** | ✅ 真实消耗 ETH |

**用户看到的关键信息（理想展示）：**

```
║                                      ║

║                                      ║
```

```
└──────────────────────────────────────┘
```

**AI Agent 辅助时的判断：**

| 场景 | 处理 |
|---|---|
| Agent 生成交易 calldata | ✅ 可以——= 读取+计算 |
| Agent 模拟交易结果 | ✅ 可以——eth_call 模拟，只读安全 |
| Agent 展示风险报告 | ✅ 可以——这是 Step 6 的 HITL 准备工作 |
| Agent 自动签名 | ❌ 绝对不可以——签名 = 授权资产变动 |
| Agent 自动广播 | ❌ 不可以——= 不可逆的链上操作 |
| Agent 追踪交易状态 | ✅ 可以——eth_getTransactionReceipt，只读 |

> 👤 **HITL：必须保留人工确认**。这是整个交互链条中最重要的 HITL 点。交易一旦广播就不可逆，而且失败的交易也会消耗 gas。Agent 的工作是做好前 5 步（解析→上下文→报价→生成→模拟），而不是替人签名。

## 设计原则：AI Agent 的钱包交互边界

```
                    Agent 可以做的              Agent 不能做的
                         │                          │
    ┌────────────────────┼────────────────────┐      │
    ▼                    ▼                    ▼      ▼
  读取数据            生成交易草稿          模拟交易   连接钱包
  查询余额            评估风险              追踪状态   切换网络
  整理展示            建议操作              记录 Trace  签名消息
                                                      发送交易
                                                      授权额度
```

**一句话规则**：Agent 可以帮你**看清**（读数据、生成计划），但不能帮你**出手**（连接、签名、发交易）。出手之前必须把风险报告交给人确认。
