## Agent Trust — 核心理解

> 信任不是一个分数，而是一组可追溯、可比较、可解释的证据。

```
Agent Identity（昨天）              Agent Trust（今天）
      │                                   │
  知道「谁是谁」                     判断「谁可信」
  Profile / Endpoint / Owner         Reputation / Stake / Slashing / Validation
```

## 知识节点

| 概念 | 一句话 | 难度 |
|---|---|---|
| **Reputation** | 历史表现信号集合——按任务类型拆开，不要压成黑盒分数 | 初级 |
| **Review** | 任务绑定评价——含任务ID、交付物、付款记录。10 个具体差评 > 100 个「good job」 | 初级 |
| **Attestation** | 可验证声明——issuer + subject + claim + evidence + expiration + revocation | 中级 |
| **Stake** | 经济抵押——让承诺有成本。但 stake ≠ 能力，资本多不代表质量高 | 中级 |
| **Slashing** | 违约罚没——需先定义证据、挑战窗口、仲裁者、误罚处理。不是安全机制就是治理风险 | 高级 |
| **Validation** | 能力验证 vs 任务结果验证——前者说「能做」，后者说「这次做好了」 | 中级 |
| **ERC-8004** | 身份+声誉+验证的 registry 标准——承载信号，不自动评分 | 高级 |

## 关键收获

1. **Agent 的可信度来自可验证行为，不是自我声明**：「我擅长合约审查」没意义。有意义的是：审过哪些、谁验证过、错过什么、有没有 stake、失败后赔没赔
2. **Reputation 要按任务类型拆开**：擅长总结治理提案的 Agent 不一定擅长写合约测试。把所有信号压成一个分数 = 用户无法判断适用场景
3. **Stake 带来选择偏差**：资本多的 Agent 不一定能力更强。小团队或公共物品 Agent 可能 stake 少但质量高。stake 应该和 validation/review/task history 一起看
4. **Slashing 的双刃剑**：自动 slashing 适合可验证违约（未按时交付、签名伪造），主观任务（报告质量）先走 dispute，不要直接 slashing
5. **时间衰减**：两年前的好表现不能完全代表今天——模型、owner、endpoint 都可能已经变了
