## 关键收获

1. **开放赛道不是"什么都可以做"，而是必须回答两个问题**：AI 具体承担了什么不可替代的工作？Web3 具体提供了什么不可替代的能力？答不清 = "AI 包装 Web3 概念"或"Web3 包装普通 AI 工具"。这是整个前沿探索的总结性判断标准——回头看前面 5 个章节，每个都通过了这个两问测试
2. **AI-native Wallet 不是钱包里加聊天框**：是让钱包理解用户目标、解释交易、管理权限、提醒风险、帮助完成多步骤链上任务。安全边界清晰——AI 解释和建议，签名/授权/高风险操作交给钱包权限系统。这和 003 实验（钱包交互地图）+ 005 设计（Guard 确定性代码）直接对齐
3. **On-chain Data Analysis Agent 的核心纪律是区分事实和推断**："某地址可能属于某机构"是推断，"资金流向某合约"是可验证事实。必须给出源链接（区块浏览器、Dune、Subgraph、RPC 结果）。这和 Bridge 的 Verifiable AI + Contract Reading Assistant 共享同一原则——结论可追溯
4. **Cross-track Combination 是 Hackathon 项目的核心方法论**：AI wallet + permission、Agentic commerce + settlement、Governance + source-preserving summary、Dev tooling + transaction interpreter、Decentralized AI + verifiable AI。关键警告——选一个核心闭环，把其他能力作为支撑层，不要一开始把所有概念都做进去
5. **新协议必须从具体摩擦出发**：现有协议为什么解决不了？需要新消息/支付/身份/证明格式吗？第一个可运行场景是什么？没有网络效应前单个项目能否获得价值？好协议从重复出现的工作流里长出来，不是"为了做协议而做协议"
6. **Research Visualization 的价值是更快发现异常、理解因果、回到证据**：合约调用图、资金流 Sankey 图、治理提案时间线、Agent 工具调用 trace、协议风险地图、模型评测对比。AI 抽取结构+生成解释，Web3 数据提供可验证来源。漂亮图不是目的

### 第一性原理

```
开放赛道的核心不是追热点，而是发现新的用户工作流。

① 先找工作流：用户现在如何完成任务，哪里慢、贵、难、不透明或不可验证
② 再拆 AI 和 Web3 职责：AI=理解/生成/规划/推荐，Web3=资产/身份/权限/记录/激励/治理
③ 最后做最小闭环：输入、处理、输出、权限、失败路径都要讲清楚
```

### 项目评估四问

```
① 用户是谁，今天怎么做这件事？
② AI 让哪一步变得明显更好？
③ Web3 让哪一步变得可验证、可拥有、可结算或可治理？
④ 最小版本是否可以在两周内做出可演示闭环？
```
