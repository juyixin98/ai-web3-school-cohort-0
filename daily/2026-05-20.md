## 关键概念

| 概念 | 一句话 |
|---|---|
| **Task Graph** | 把目标拆成节点+依赖，每步有输入/输出/权限/停止条件 |
| **State Machine** | 用显式状态替代聊天历史，系统不会「忘记自己在哪里」 |
| **Human-in-the-loop** | 只读自动、小额 session key、高风险必须人工确认 |
| **Retry/Fallback** | 读可重试、写先判断是否已广播、pending 不能再发 |
| **Trace** | 每步输入/判断/调用/结果都有记录，才能复盘 |
| **Regression Set** | 固定用例（正常+错误链+无限 approve）防止安全性退化 |

## 关键收获

1. Agent Workflow 的本质是把概率模型放进确定性流程——模型可以推理，但系统必须有边界和停止条件
2. Human-in-the-loop 的分层策略：**低风险操作可以自动授权执行，高风险和自动化风险必须在关键节点由人管控**。不是每个步骤都让人确认（太慢），而是把确认点放在不可逆的资产变动上
3. 链上 Agent 的 Retry 不能照搬 Web2 经验：读失败可重试，写失败要先判断是否已广播，pending 状态绝不能盲目再发交易（可能导致双花）
4. 没有 Trace 的 Agent 出问题只能看聊天记录；有 Trace 才能复盘是模型理解错、工具返回错、policy 漏了、还是用户确认了不该确认的动作
