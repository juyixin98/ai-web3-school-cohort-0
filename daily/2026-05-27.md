## 验证强度分层

```
输出影响              验证方案              成本
────────────────────────────────────────────────────
日常复盘/查问题     → Audit Trail         极低（记日志）
服务来源证明        → 签名日志            低
隐私数据评分        → TEE attestation     中（硬件信任）
链上关键判断        → ZK proof            高（密码学）
LLM 完整推理        → 暂不现实            极高
```

## 关键收获

1. **验证成本必须和输出影响力匹配**：总结新闻不需要 ZK proof，释放 escrow 不能只靠一句模型输出。大多数场景 Audit Trail + Attestation 就够用，不要迷信 ZK
2. **TEE 不是纯密码学信任**：它依赖硬件和供应链信任假设。文档里要写清楚信任谁、证明什么、不证明什么——不要把它包装成「绝对安全」
3. **LLM 完整 ZK 证明目前不现实**：实用方案是混合——只证明关键步骤（小模型分类、后处理规则），复杂推理走 TEE + 签名日志 + challenge
4. **Proof of Inference 先明确目标再选技术**：「证明输入没变」和「证明完整计算正确」是两个完全不同的目标，成本差几个数量级
5. **Audit Trail 是最容易被跳过但最重要的基础层**：即使没有 TEE/ZK，完整记录输入/输出/模型版本/工具调用/用户确认/交易哈希，也能支持复盘、争议和改进
