## 关键收获

1. **Agent 不应该拥有"钱包"，它只应该拥有一组可限制、可审计、可撤销的能力**：把私钥交给 Agent 是最脆弱的设计。正确方式是——钱包/智能账户持有资产和执行权，Agent 只提出意图、生成候选交易、请求有限权限、在规则允许时触发动作。这与 005 的 Smart Account + Session Key 设计完全一致
2. **Permission Policy 的 7 维度比 005 设计更完整**：资产范围 + 金额上限 + 目标合约 + 函数范围 + 价格/滑点 + 时间窗口 + 频率限制。005 做了金额、时间、白名单，但缺了函数范围（只允许 swap 不允许 approve）、价格/滑点阈值和频率限制
3. **AI Wallet UX 的重点不是"少点几次确认"，而是让确认有意义**：用户要看到的不只是 calldata，而是——资产变化、授权对象、最大损失、失败路径。界面要展示目标任务、行动计划、权限请求、交易解释、风险提示、撤销入口。这验证了 002 确认面板的 6 字段设计
4. **Pre-transaction Simulation 是 Agent 钱包的强制动作，不是可选功能**：模拟要回答——token 余额变化、授权产生、合约调用、revert 风险、价格/滑点/手续费、无限授权风险。关键字段来自结构化解析和链上模拟，不能只靠自然语言总结
5. **产品应该按风险逐步开放**：只读 → 生成草稿需人工签名 → 小额白名单短期 session key → 复杂自动化带 guard+审计+撤销。这四阶段路径可以直接作为 Hackathon 产品的功能 roadmap
6. **Recovery / Revocation 是"默认能力"而非"后期补救"**：用户要能在一个地方看到所有 Agent/Key、权限范围、已用额度、过期时间、最近执行、暂停/撤销/恢复入口。高价值场景还需要 guardian、多签、延迟执行、社交恢复
