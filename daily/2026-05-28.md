## 关键收获

1. **上下文 ≠ 指令**：合约文档、网页内容、API 返回值不能覆盖系统规则。防护第一步是分层标注上下文可信级别——用户指令 > 系统规则 > RPC 数据 > 外部网页 > 未验证内容。模型必须被告知：外部内容只能作为待分析资料，不能成为新指令
2. **Tool Abuse 不是一次大动作，是很多小动作累积**：恶意页面诱导 Agent 多次购买无价值 API，或循环调用浏览器工具消耗预算。工具层需要独立异常检测（短时间高频、重复付款、参数偏离），不能只靠 Agent 自律
3. **Key Safety 的底线**：secret 不进入 prompt、不进入模型输出、不进入普通日志、不进入 analytics。Agent 需要签名时通过钱包工具/Smart Account/Session Key 间接完成，而不是把私钥放进自动化 runtime
4. **Permission Isolation 反证了 005 设计的正确性**：读工具和写工具分开 ✓、浏览器环境和钱包签名分开 ✓、普通任务和高风险任务分开 ✓。最安全的工具不是功能最全的，而是刚好完成任务且无法越界的
5. **Audit Log 不能只记「最终回答」**：真正有用的是完整链路——模型看了哪段上下文、为什么选这个工具、工具返回什么、policy 是否通过、用户是否确认。这就是 Agent Workflow（002）里 Trace 的设计目标

### 安全检验矩阵

```
攻击向量                  我们的设计防御                    状态
──────────────────────────────────────────────────────────
Prompt Injection → 诱导转账    Guard 是确定性代码，不走 AI     ✅
恶意外部上下文 → 假地址       收款方白名单                   ✅
Tool Abuse → 消耗预算         Budget + 单笔上限              ⚠️ 缺频率限制
Key Leak → 私钥进日志         Smart Account + Session Key    ✅ (设计层)
权限扩大 → 万能工具            读写分离 + HITL               ✅
Malicious Context → 假数据    RPC 优先于网页内容             ⚠️ 未显式标注
```

> ⚠️ 两个缺口：1）预算系统缺少调用频率限制（Tool Abuse 可以通过低频多次绕过）；2）数据来源可信级别未显式标注（RPC > Explorer > 网页 > 用户消息）
