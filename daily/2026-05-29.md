## 关键收获

1. **隐私问题的本质不是单点泄漏，是拼接**：链上地址本身是公开的，但地址 + 交易历史 + 聊天记录 + Agent 偏好 + 邮箱 = 完整用户画像。AI 系统扩大了数据面（读更多上下文、保留记忆、调第三方模型），隐私设计必须从架构第一天进入
2. **Data Boundary 要按工具划分，不能按系统划分**：浏览器工具、钱包工具、模型工具、索引工具看到的数据不应默认互通。每个数据字段都要问：是否必要、是否可脱敏、是否可本地处理、是否会进日志、是否会发给第三方模型。最好的 Data Boundary 是数据流图，不是隐私政策长文
3. **Secret Management 与 AI Security Key Safety 高度一致**：私钥/API key/session token/JWT 不进 prompt、不进模型输出、不进普通日志、不进 analytics。Agent 通过受控工具间接使用 secret。本次学习验证了 005 设计——Smart Account + Session Key 就是这个原则的实现。额外收获：系统应识别并拒绝处理用户不小心粘贴的助记词
4. **Local AI 的务实定位是做隐私网关，不做主力推理**：先在本地筛选和脱敏（如"与本任务有关的 3 笔交易"），再把必要摘要发给强模型。本地不等于绝对安全——仍需 sandbox 和权限控制
5. **Private Memory 不等于不可见黑箱**：用户要知道记了什么、为什么记、如何删除、是否用于训练。常见风险是"为了更智能而永久保存一切"——默认应该最小化、可查看、可删除。分层方案：短期任务记忆 / 长期偏好 / 敏感身份信息 / 钱包记录，不同层不同保留时间和访问权限
6. **Minimal Disclosure 最实用的形式是"摘要代替原始数据"**：模型不一定需要全部聊天记录，只需要和当前任务相关的几条结构化事实。更高级的方案：ZK 证明余额够付而不暴露持仓、一次性地址、按应用隔离身份
7. **User Consent 应该具体到动作和数据类型**："允许读取当前地址余额"和"允许分析最近 30 天交易"应该是不同开关。Consent 可撤销，且撤销后要说明哪些删除、哪些链上无法删除、哪些因合规保留
