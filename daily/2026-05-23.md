## 为什么需要 Escrow

Machine Payment 解决了「Agent 怎么花钱」，但付款和交付之间有时间差：先付可能被骗，先交付可能收不到钱。Escrow 在中间加一层——资金先锁定，服务确认后释放，失败则退款。

```
Machine Payment（昨天）         Settlement & Escrow（今天）
      │                              │
  怎么付款                           付了之后呢？
  Budget → Quote → Pay               Lock → Deliver → Accept → Release
                                          ↘ Fail → Refund → Dispute
```

## 关键概念

| 概念 | 一句话 | 难度 |
|---|---|---|
| **Escrow** | 资金先锁在合约里，交付条件满足后释放 | 初级 |
| **Receipt** | 收据≠"已付款"，需记录谁/多少/任务ID/txHash/验收状态 | 初级 |
| **Delivery Proof** | 证明服务方确实交付了结果（hash/日志/签名/TEE attestation） | 中级 |
| **Acceptance** | 验收标准要拆成可检查条件，别只靠模型一句"看起来完成了" | 中级 |
| **Refund** | 退款规则在任务开始前就写清——不是失败后的临时善意 | 初级 |
| **Dispute** | 争议流程可预期/可记录/可执行，发起成本要适中 | 高级 |
| **Evaluator** | 判断交付是否合格的角色——AI初审+人工复核+challenge window | 高级 |
| **ERC-8183** | Agent commerce 标准草案：任务+状态+proof+settlement+dispute | 高级 |

## 关键收获

1. **Escrow 的核心是状态机，不是锁钱合约**：Created→Funded→Delivered→Accepted→Released，每个状态要规定谁能触发、需要什么证据、超时怎么处理
2. **先定义业务流程，再定义资金流**：如果没有任务描述+交付标准+争议路径，锁钱只会把双方困在合约里
3. **Delivery Proof ≠ "我发了"**：证明必须和原始任务对应——hash、时间戳、版本信息，缺一不可
4. **AI evaluator 可以初审，但不能终审高风险任务**：建议「AI 初审 + challenge window + 人工复核」组合。evaluator 本身的错误率也需要被评估
5. **部分交付需要部分退款规则**：服务方完成了数据抓取但分析失败，是否按比例付款？没有这个规则，双方容易进入 dispute
