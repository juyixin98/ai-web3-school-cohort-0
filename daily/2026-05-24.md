## Agent Identity — 核心理解

> Agent Identity 不是给 Agent 起个名字，而是让用户、服务和其他 Agent 能验证：它是谁、谁控制它、能做什么、入口在哪、历史能不能追溯。

```
之前的章节                              本章位置
┌─────────────────────────┐            ┌─────────────────┐
│ Wallet → Payment → Escrow│  ──→      │ Agent Identity  │
│ (能花钱/能结算)          │  付款前    │ (知道付给谁)     │
└─────────────────────────┘  必须先    └─────────────────┘
```

## 知识节点

| 概念 | 一句话 | 难度 |
|---|---|---|
| **Agent Profile** | 公开说明文件——人和机器都能读：名称/能力/价格/endpoint/owner/版本历史 | 初级 |
| **Capability** | 能力声明要绑定 schema+价格+限制+风险等级。不是「我什么都能做」 | 中级 |
| **Service Endpoint** | 调用入口（HTTPS/A2A/MCP）。endpoint 被劫持=身份作废，更新需 owner 签名 | 初级 |
| **Registry** | 链上/链下身份登记——能证明「谁注册的」，不能证明「一定好用」 | 中级 |
| **DID / VC** | 去中心化身份+可验证声明——任何人能签发≠任何声明可信，关键看 issuer | 中级 |
| **A2A** | Agent 间通信协议——身份告诉你「和谁说话」，A2A 负责「怎么协作」 | 中级 |
| **Ownership** | 谁控制 profile/收款地址/endpoint 更新。高价值 Agent 不应由单签热钱包控制 | 高级 |

## 关键收获

1. **Agent 从临时会话升级为经济参与者**：没有身份，今天的「高质量 Agent」明天可能换服务端、换 owner、换能力描述，用户无法判断
2. **Profile 更新不能静默发生**：换模型、换收款地址、增加高风险能力——每一次变更都应该是信任信号，而不是隐藏动作
3. **Registry ≠ 信任层**：Registry 证明「这个身份是谁注册的」，不能证明「这个 Agent 安全好用」。能力和声誉需要后续验证（下一章 Agent Trust）
4. **operator ≠ owner**：运行服务的人和掌控身份更新/收款地址的人应该可以分离——这是责任归属的基础

### 与前序章节的连接

```
Agent Wallet → Machine Payment → Settlement & Escrow → Agent Identity → Agent Trust
    │               │                   │                   │               │
  权限边界      怎么花钱           付后怎么结算        付给谁？         可信吗？
```

Agent Identity 是所有支付和委托的前置条件——不知道对方是谁，后面的 Budget/Quote/Escrow 都建立在一个脆弱的假设上。
