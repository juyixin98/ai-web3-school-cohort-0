## 关键收获

1. **Agentic Commerce 的本质是把 Bridge 的分散模块串成商业闭环**：服务发现 + 报价 + 预算控制 + 付款 + 验收 + 收据 + 争议 = 一个完整的 Agent 商业交易。这不是新概念，而是 Agent Wallet / Machine Payment / Settlement / AI Oracle / Verifiable AI 在真实商业场景中的组合
2. **Payment Intent 是用户授权到机器约束的翻译层**：不能是"帮我买最合适的"，必须是结构化的——目标、预算（单次/日/总额）、资产、白名单、质量要求、退款条件、有效期。Agent 可以在边界内决策，但不能把边界解释得越来越宽
3. **Budget Control 的分层比 005 设计更细**：任务预算 + 服务预算 + 时间预算 + 风险预算 + 失败预算。005 做了单笔上限和日预算，缺了时间维度（每小时限制）和失败预算（连续失败 N 次自动停止）
4. **Proof of Task Completion 按服务类型区分证据**：API 服务看请求 ID + 响应 + 时间戳，推理服务看模型版本 + 输入摘要 + 输出哈希，执行服务看交易哈希 + 事件日志。与 Verifiable AI 一致——证据强度匹配交易价值，低价值日志够用，高价值需要链上 receipt
5. **Escrow Flow = Settlement & Escrow 章节的直接实现**：同样的状态机（创建→托管→交付→验收→放款/退款/争议），但加上了一条 Agentic Commerce 特有的警示——高价值或主观结果不应该只靠模型自动放款
6. **On-chain Receipt 不存完整内容，存可核对证据**：payer/provider/task id + 金额 + 输出哈希 + 状态。隐私数据只存哈希或加密引用——呼应 AI Privacy 的 Minimal Disclosure
